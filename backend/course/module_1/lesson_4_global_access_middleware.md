## Lesson 4: Global Access Control with Middleware

In the previous lesson, we explored Django's authentication and authorization system, focusing on how to protect individual views and specific objects. This lesson will delve into how to leverage middleware for global access control, expanding on the concepts introduced in `middleware.md`.

### 1. What is Django Middleware?

As we briefly touched upon, middleware is a framework of hooks into Django’s request/response processing. It acts like a series of "filters" or "layers" that every HTTP request passes through on its way to your view, and then again as the HTTP response leaves your view.

**The Request/Response Flow with Middleware:**

```
Browser Request
    ↓
Middleware Layer 1 (e.g., SecurityMiddleware)
    ↓
Middleware Layer 2 (e.g., SessionMiddleware)
    ↓
Middleware Layer 3 (e.g., AuthenticationMiddleware)
    ↓
YOUR CUSTOM MIDDLEWARE
    ↓
URL Router (`urls.py`)
    ↓
View Function/Class
    ↓
(Response generated by View)
    ↑
YOUR CUSTOM MIDDLEWARE (again, for response processing)
    ↑
Middleware Layer 3 (e.g., AuthenticationMiddleware)
    ↑
Middleware Layer 2 (e.g., SessionMiddleware)
    ↑
Middleware Layer 1 (e.g., SecurityMiddleware)
    ↓
Browser Response
```

**Key Characteristics of Middleware:**

*   **Global Impact:** Middleware processes *every* request (and potentially every response) that hits your Django application.
*   **Order Matters:** The order in which middleware is listed in `settings.MIDDLEWARE` is crucial. Middleware listed earlier processes requests first and responses last.
*   **`get_response` Callable:** A middleware typically receives a `get_response` callable during initialization. Calling `self.get_response(request)` passes the request further down the chain towards the view.
*   **Modifying Request/Response:** Middleware can modify the `request` object before it reaches the view, or the `response` object before it's sent back to the client. It can also short-circuit the request by returning an `HttpResponse` directly, preventing the request from ever reaching the view.

### 2. When to Use Middleware for Authorization

Middleware is ideal for **broad, cross-cutting concerns** that apply across multiple views or your entire application.

**Good Use Cases for Authorization Middleware:**

*   **Global Login Requirement:** Force all non-public pages to require authentication.
*   **Role-Based URL Restriction:** Restrict access to entire URL prefixes (e.g., `/staff/`, `/admin-panel/`) based on user roles (`is_staff`, group membership).
*   **Account Status Checks:** Ensure a user's account is active, verified, or has a certain subscription level before accessing most features.
*   **Multi-Tenancy:** In applications serving multiple organizations, ensure a user can only access data belonging to their assigned organization based on URL or subdomain.
*   **IP Restriction:** Allowing access only from specific IP addresses.

**When NOT to use Middleware for Authorization:**

*   **Object-Level Permissions:** Middleware runs *before* the view determines the specific object being requested. Logic like "users can only edit their *own* posts" belongs in the view (or with tools like Django Guardian), not in middleware.
*   **Complex View-Specific Business Logic:** Keep detailed business rules within the views or services.
*   **Heavy Database Queries:** Since middleware runs on every request, avoid computationally expensive operations or excessive database lookups. Cache results if necessary.

### 3. Creating Custom Authorization Middleware

Let's walk through the process of building a custom middleware, as demonstrated in `middleware.md`.

#### Step 1: Create the Middleware File

It's good practice to create a `middleware.py` file inside one of your Django apps (e.g., `blog/middleware.py`).

#### Step 2: Write the Middleware Class

A middleware class typically looks like this:

```python
# blog/middleware.py
from django.shortcuts import redirect
from django.urls import reverse
from django.http import HttpResponseForbidden

class SimpleAuthMiddleware:
    def __init__(self, get_response):
        # One-time configuration and initialization.
        self.get_response = get_response

    def __call__(self, request):
        # Code to be executed on each request BEFORE the view is called.
        
        # Example: Log all incoming requests
        print(f"Incoming request path: {request.path}")

        # Pass the request down the middleware chain to the view
        response = self.get_response(request)

        # Code to be executed on each response AFTER the view is called.
        # Example: Add a custom header to all responses
        # response['X-My-Custom-Header'] = 'Processed by SimpleAuthMiddleware'
        
        return response
```

#### Step 3: Register the Middleware in `settings.py`

You must add the full Python path to your middleware class in the `MIDDLEWARE` list in your project's `settings.py`. **The order is critical!**

```python
# auth_project/settings.py
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware', # request.user is established here
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
    # Add your custom middleware AFTER AuthenticationMiddleware
    'blog.middleware.SimpleAuthMiddleware', 
]
```
**Why order matters:** Your custom authorization middleware will likely rely on `request.user` being available and correctly populated. `django.contrib.auth.middleware.AuthenticationMiddleware` is responsible for this, so your middleware must be placed *after* it.

### 4. Practical Example: Global Login Required Middleware

Let's implement a common scenario: requiring all users to be logged in to access most of the site, with a few exceptions (login page, admin site, and a hypothetical public `about` page). This is directly from `middleware.md`'s example.

```python
# blog/middleware.py
from django.shortcuts import redirect
from django.urls import reverse, resolve, Resolver404 # resolve for getting URL name
from django.conf import settings # To access LOGIN_URL

class LoginRequiredMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response
        self.login_url = settings.LOGIN_URL # Defined in settings.py (e.g., '/accounts/login/')
        self.admin_url_prefix = '/admin/'

        # List of URL names that should be publicly accessible
        self.public_url_names = ['login', 'signup', 'about'] # Assuming these URL names exist

    def __call__(self, request):
        # --- Check for public paths ---
        is_public_path = False
        # Check if it's the login URL itself or an admin path
        if request.path == self.login_url or request.path.startswith(self.admin_url_prefix):
            is_public_path = True
        else:
            # Try to resolve the URL name to check against public_url_names
            try:
                resolver_match = resolve(request.path_info)
                if resolver_match.url_name in self.public_url_names:
                    is_public_path = True
            except Resolver404:
                # If URL doesn't resolve, it's likely not a named public path,
                # but we'll let it pass for now if it's not authenticated.
                # A more strict approach might deny it.
                pass

        # --- Enforce login for non-public paths ---
        if not request.user.is_authenticated and not is_public_path:
            # Redirect to login page, preserving the original requested path
            # so the user can be redirected back after successful login.
            return redirect(f"{self.login_url}?next={request.path}")

        # If authenticated, or if it's a public path, proceed to the next middleware/view
        response = self.get_response(request)
        return response
```

**To make this work, you'd need:**

1.  **`settings.py`:**
    ```python
    LOGIN_URL = '/accounts/login/'
    LOGIN_REDIRECT_URL = '/' # Where to go after successful login
    LOGOUT_REDIRECT_URL = '/accounts/login/' # Where to go after logout
    ```
2.  **`urls.py`:** Ensure you have URL patterns with the names `'login'`, `'signup'`, and `'about'` (if applicable).
    ```python
    # auth_project/urls.py
    from django.contrib import admin
    from django.urls import path, include
    from blog.views import about_view # Assuming you have an about_view

    urlpatterns = [
        path('admin/', admin.site.urls),
        path('accounts/', include('django.contrib.auth.urls')), # Provides 'login', 'logout' etc.
        path('about/', about_view, name='about'), # Our public about page
        path('', include('blog.urls')), # Other app URLs
    ]
    ```
3.  **`blog/views.py`:** A simple `about_view`.
    ```python
    # blog/views.py
    from django.shortcuts import render

    def about_view(request):
        return render(request, 'blog/about.html')
    ```

### 5. Common Middleware Mistakes

*   **Incorrect Order:** Placing your authorization middleware before `AuthenticationMiddleware` will mean `request.user` is not available.
*   **Heavy Operations:** Don't perform slow database queries or complex calculations in middleware, as they will affect performance for *every* request.
*   **Forgetting Exclusions:** Always ensure your login page, signup page, and any other truly public endpoints are explicitly excluded from "login required" middleware, otherwise you'll create an infinite redirect loop.
*   **Object-Level Checks:** As mentioned, avoid trying to enforce object-specific permissions here.
